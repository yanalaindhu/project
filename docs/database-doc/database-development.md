# TRIVARNA - Day 1 Database Work Documentation

## Date

Day 1

## Module

Database Development (Supabase)

---

# Objective

The objective of Day 1 was to design and implement the database foundation for the TRIVARNA system.

The database will store user information, wellness data, schedules, goals, habits, AI insights, and burnout prediction data required by the platform.

---

# Tasks Completed

## 1. Database Schema Design

Designed the database structure required for the TRIVARNA project.

The schema was created based on the project requirements and future AI modules.

---

## 2. Tables Created

### profiles

Stores user information.

Fields:

* id
* full_name
* age
* occupation
* created_at

---

### daily_checkins

Stores daily wellness information.

Fields:

* mood_score
* stress_level
* energy_level
* sleep_hours
* water_intake
* exercise_minutes
* productivity_score
* notes

Purpose:

Used to understand the user's daily condition.

---

### trivarna_scores

Stores calculated wellness scores.

Fields:

* mind_score
* body_score
* lifestyle_score
* overall_score

Purpose:

Represents the overall life balance of the user.

---

### burnout_predictions

Stores burnout analysis results.

Fields:

* burnout_score
* risk_level

Purpose:

Used to predict future burnout risk.

---

### goals

Stores user goals.

Fields:

* goal_name
* category
* target_date
* progress_percentage
* status

Purpose:

Tracks goal progress.

---

### habits

Stores user habits.

Fields:

* habit_name
* category
* target_frequency

Purpose:

Tracks healthy habits.

---

### habit_logs

Stores habit completion records.

Fields:

* completed
* completed_date

Purpose:

Tracks habit consistency.

---

### schedules

Stores AI-generated schedules.

Fields:

* schedule_date
* generated_by_ai

Purpose:

Stores daily plans generated by the system.

---

### schedule_tasks

Stores tasks inside schedules.

Fields:

* task_name
* category
* start_time
* end_time
* status

Purpose:

Stores detailed schedule activities.

---

# Database Relationships

Implemented relationships between tables using foreign keys.

Relationship Structure:

profiles
├── daily_checkins
├── trivarna_scores
├── burnout_predictions
├── goals
├── habits
│   └── habit_logs
└── schedules
└── schedule_tasks

---

# Security Planning

Planned Row Level Security (RLS) implementation.

Purpose:

* Protect user data
* Ensure users can access only their own records
* Improve application security

Example:

User A cannot access User B's data.
